## 2024-05-24 - Do not rely on suppressed `chmod` to set directory permissions
**Vulnerability:** In `qwen3_embed/common/model_management.py` and `qwen3_embed/common/utils.py`, `mkdir` was called without a `mode` parameter. After directory creation, `os.chmod` or `Path.chmod` calls were used to set secure permissions (`0o700`), but these were wrapped in `contextlib.suppress(OSError)`. Because `chmod` failures were suppressed (to avoid functional regressions on systems where `chmod` unsupported, e.g. Windows/NFS), the directories could silently end up with insecure default permissions on these systems. Furthermore, setting the mode after directory creation introduces a Time-of-Check to Time-of-Use (TOCTOU) race condition window where the directory exists with insecure permissions.
**Learning:** Silently masking permission failures with `contextlib.suppress(OSError)` leaves directories vulnerable on multi-user systems. However, unconditionally removing `contextlib.suppress(OSError)` introduces functional regressions (crashes on systems where `chmod` fails).
**Prevention:** Always enforce secure permissions when creating cache or download directories by passing `mode=0o700` directly to `mkdir`. The `chmod` with `contextlib.suppress(OSError)` can still be kept as an extra precaution, but `mode=0o700` in `mkdir` guarantees that the directory is created securely in a single atomic step without a TOCTOU race condition, and respects the OS's native permission implementation at creation time.
